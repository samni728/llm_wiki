# 企业内部 Wiki 规划方案

## 背景

当前 `feishu-workspaces` 项目已经具备企业知识收集器的核心能力：

- 通过飞书入口接收群聊、私聊、附件和机器人查询。
- 将图片、音频、PDF、Markdown 等内容解析为可检索文本。
- 使用 SQLite 保存消息、附件、解析结果、权限、审计和任务状态。
- 使用 LanceDB/向量索引支持 RAG 检索。
- 通过 API、Web UI 和飞书 Bot 提供查询与回答能力。
- 支持本地化部署，适合 Mac mini + NAS + 本地模型的中小企业环境。

`llm_wiki` 项目的核心价值不是替代现有 RAG，而是提供一种“知识编译器”模式：LLM 将原始资料持续整理成结构化、可维护、可复审、可追溯的 Markdown Wiki。

当前实施边界：

> 飞书 / OCR / MinerU 是主知识入口，SQLite + LanceDB 是主 RAG 检索层，企业 Wiki 是人工确认后的单向沉淀层。当前阶段 Wiki 不回写飞书 chat，也不反向进入主 RAG。

因此，企业版 Wiki 不应该直接替代现有项目，而应该与现有项目共生：

```text
飞书业务流
  -> 现有收集器：归档、解析、权限、检索、审计
  -> 企业 Wiki 引擎：提炼、候选、审核、发布、维护
  -> 管理层/主管使用的组织知识库
```

## 定位

企业版 Wiki 的定位不是“全员开放的聊天记录搜索”，而是：

> 飞书业务流驱动的、本地化、带审核机制的企业经营 Wiki。

它面向管理层、主管、负责人等被授权角色，目标是把每天散落在飞书里的业务信息，沉淀成可长期复用的组织知识。

核心原则：

- 原始事实留在现有数据库中，Wiki 保存经过提炼和审核的知识表达。
- LLM 负责起草、分类、补链、检查和维护建议。
- 人类负责人负责确认、发布、修正和敏感内容处理。
- 正式 Wiki 内容必须可追溯到来源消息、附件、解析结果或人工备注。
- 重大更新不直接覆盖，应保留变更记录或版本线索。

## 对 `llm_wiki` 的理解校正

`llm_wiki` 不是一个“多人设自治 Agent 系统”，而是一个以项目目录为中心的知识维护工作台。

它的实现更接近：

- 一个 Markdown Wiki 项目。
- 一组固定知识动作：`ingest`、`query`、`lint`、`review`、`cleanup`。
- 针对不同任务使用不同 prompt，而不是让多个独立人格自由运行。
- 以文件系统作为知识表达层，以 `.llm-wiki/` 保存部分应用状态。

企业版可以吸收它的核心方法，但不建议完整照搬桌面应用形态。

适合复用的思想：

- 原始资料与 Wiki 分层。
- 双阶段摄入：先分析，再生成。
- `schema.md` / `purpose.md` 驱动 LLM 行为。
- `index.md` / `overview.md` / `log.md` 辅助导航和维护。
- `sources` 字段追溯来源。
- review 队列、lint 检查、过期与冲突提示。
- Markdown 文件作为正式知识资产。

不建议直接照搬的部分：

- 个人桌面项目模型。
- 将 review/chat 状态主要保存在 `.llm-wiki/*.json`。
- 单项目、单用户的 UI 和最近项目状态。
- 面向个人文档的默认 prompt。

## 共生架构

建议采用四层架构。

### 1. 采集层

由现有飞书知识库项目继续负责。

职责：

- 接收飞书消息和附件。
- 下载并保存原始文件。
- OCR/STT/PDF/Markdown 解析。
- 保存消息、附件、解析内容、chunk、向量、权限和审计。
- 提供 `/api/query`、`/api/answer` 等能力。

这一层是事实底座，不应该被 Wiki 覆盖。

### 2. 候选沉淀层

每天或按固定周期，由 LLM 读取新增聊天和解析内容，提炼值得沉淀的候选知识。

职责：

- 按群、业务域、关键词、时间窗口读取新增内容。
- 使用对应业务 prompt 进行提炼。
- 生成候选知识卡片，而不是直接写入正式 Wiki。
- 标记来源、置信度、敏感风险、推荐页面类型和推荐动作。

示例流程：

```text
每日群聊记录
  -> 按群/业务域分组
  -> 使用财务/仓库/客户/项目等 prompt 提炼
  -> 生成候选知识卡片
  -> 进入审核队列
```

### 3. 审核发布层

由负责人、主管或管理层定期批量处理候选知识。

职责：

- 批准发布。
- 退回重写。
- 合并到已有页面。
- 标记敏感并剔除。
- 标记为待确认。

原则：

- LLM 不直接拥有正式发布权。
- 人类不需要重写大段内容，只做判断和少量修正。
- 重大改动进入复审队列。

### 4. 正式 Wiki 层

正式 Wiki 保存审核通过的长期知识。

职责：

- 保存当前有效结论。
- 保留来源链。
- 维护页面间链接。
- 记录变更历史。
- 支持 lint、补链、冲突检查和过时提醒。

建议正式 Wiki 仍使用 Markdown 文件保存，便于 Obsidian、Git、备份、人工查看和后续迁移。

## 数据库复用策略

现有数据库应该复用，而且应该作为企业 Wiki 的事实源。

建议分工：

```text
SQLite
  - 原始消息
  - 附件与解析结果
  - chunk 元数据
  - 权限与审计
  - 候选知识卡片
  - 审核记录
  - 发布记录
  - 定时任务状态

LanceDB / 向量库
  - 原始内容向量
  - 当前阶段不写入 Wiki 页面向量
  - 当前阶段不写入候选知识向量

Markdown Wiki
  - 正式知识页面
  - index.md
  - overview.md
  - log.md
  - schema.md
  - purpose.md
```

不建议把正式 Wiki 只存在数据库里。数据库适合事实、状态和查询；Markdown 适合长期知识表达、版本管理和人工维护。

当前阶段的向量边界：

- 主 LanceDB 只索引飞书、OCR、MinerU、PDF/STT 等原始或解析来源。
- Wiki 草稿和 Wiki 页面暂不写入主 `KnowledgeChunk` 或主 LanceDB。
- 如未来需要检索 Wiki，应设计独立的 `wiki_search` 或独立向量库，先通过产品评审，避免 RAG 答案被沉淀后又被主 RAG 当作原始证据引用。

## LLM 复用策略

LLM 可以复用现有项目的本地模型或 OpenAI-compatible 接口。

可复用部分：

- provider / endpoint / api key / model 配置。
- 本地模型服务。
- embedding 与 reranker。
- prompt 模板管理方式。
- 后台任务队列与定时调度。

需要重写的部分：

- 面向企业业务流的抽取 prompt。
- 面向不同群类型的业务 prompt。
- 面向不同页面类型的写入 prompt。
- 敏感信息识别与剔除 prompt。
- 审核辅助 prompt。
- 版本变更与过期检测 prompt。

推荐不要使用单一通用 prompt 处理所有群聊，而是采用两层 prompt 体系。

### 群/业务域 Prompt

决定“从这类群聊中抽取什么”。

示例：

- 财务群：付款、回款、发票、对账、成本异常、财务风险。
- 仓库群：库存异常、出入库问题、物流时效、补货策略、错发漏发。
- 客户群：客户需求、投诉、报价变化、订单异常、合作状态变化。
- 项目群：进度、风险、决策、负责人变化、里程碑。
- 管理群：跨部门决策、经营风险、制度调整、重点事项。

### 知识对象 Prompt

决定“抽出来后如何写入 Wiki”。

示例：

- 决策页写入 prompt。
- 流程页更新 prompt。
- 客户/供应商页更新 prompt。
- 异常事件页写入 prompt。
- FAQ 页生成 prompt。
- 周报/月报综合 prompt。

## 信息进入规则

不应该把每日聊天原样写入 Wiki，而应该由 LLM 先提炼成候选知识。

适合进入候选池的内容：

- 明确决策。
- 项目状态变化。
- 客户或供应商关键变化。
- 业务异常与处理结果。
- 可复用经验。
- 流程调整。
- 高频问答。
- 风险信号。
- 会议结论和行动项。

默认忽略：

- 日常闲聊。
- 单纯情绪表达。
- 没有上下文的碎片。
- 重复转发。
- 一次性通知。
- 未确认猜测。

默认剔除或隔离：

- 工资和绩效。
- 个人隐私。
- 银行卡、身份证、手机号等敏感数据。
- 高敏感财务明细。
- 法务、人事等高风险信息。

## 候选知识卡片

LLM 每日提炼的结果建议先保存为结构化候选卡片。

示例结构：

```json
{
  "date": "2026-04-24",
  "source_chat_id": "oc_xxx",
  "business_domain": "warehouse",
  "type": "issue",
  "title": "某客户订单发货延迟",
  "summary": "当天讨论显示该订单延迟与仓库拣货和物流交接有关，需要复盘。",
  "recommended_action": "create_page",
  "recommended_wiki_path": "wiki/issues/customer-shipping-delay.md",
  "confidence": 0.74,
  "sensitivity": "normal",
  "review_status": "pending",
  "sources": [
    {
      "kind": "message",
      "message_id": "om_xxx",
      "chat_id": "oc_xxx",
      "timestamp": "2026-04-24T10:20:00+08:00"
    }
  ]
}
```

候选卡片不是正式知识，只是 LLM 给人的“待处理草稿”。

## Wiki 页面类型

第一版建议控制页面类型，不要过度发散。

推荐目录：

```text
wiki/
  index.md
  overview.md
  log.md
  projects/
  decisions/
  processes/
  stakeholders/
  meetings/
  issues/
  faq/
```

页面类型：

- `projects/`：项目、阶段、负责人、风险、里程碑。
- `decisions/`：管理决策、背景、影响范围、生效时间。
- `processes/`：发货、采购、对账、售后、仓储、审批等流程。
- `stakeholders/`：客户、供应商、部门、关键角色。
- `meetings/`：重要会议纪要、结论、行动项。
- `issues/`：异常事件、原因、处理、复盘、预防。
- `faq/`：高频问题和标准回答。

建议 frontmatter：

```yaml
---
type: issue
title: 某客户订单发货延迟
status: active
visibility: management
sensitivity: normal
owners: []
review_status: approved
sources: []
created: 2026-04-24
updated: 2026-04-24
version: 1
---
```

## 审核流

第一版审核动作建议保持简单。

候选卡片状态：

```text
pending
  -> approved
  -> rejected
  -> needs_rewrite
  -> merged
  -> sensitive_excluded
  -> needs_confirmation
```

审核动作：

- 批准发布。
- 退回重写。
- 合并到已有页面。
- 标记敏感并剔除。
- 标记待确认。

审核节奏：

- 每日：决策、异常、项目状态变化。
- 每周：FAQ、流程优化、客户/供应商状态整理。
- 每月：页面合并、过期检查、版本归档。

## 当前 LLM Wiki 运行链路

当前 `llm_wiki` 的实际工作流已经可以支撑“外贸电商”等单业务域 Wiki 的基础闭环。

以“外贸电商运营知识库”为例：

```text
外贸电商资料
  -> raw/sources/ 原始资料池
  -> 用户点击 ingest / 提取到 Wiki
  -> LLM 分析资料
  -> LLM 生成结构化 Markdown 页面
  -> 写入 wiki/
  -> 更新 index.md、overview.md、log.md
  -> 生成 review items
  -> Chat 基于 wiki/ 页面进行问答
```

其中 `raw/sources/` 只表示“资料已上传”，不等于“知识已入库”。只有经过 `ingest` 后，资料才会被提炼为正式 Wiki 页面。

当前 `ingest` 是双阶段流程：

1. 分析阶段：LLM 阅读原始资料，提取实体、概念、主要观点、和现有 Wiki 的关联、潜在冲突与补充建议。
2. 生成阶段：LLM 根据分析结果输出多个 `FILE` block，由系统写入 `wiki/` 目录，并生成必要的 `REVIEW` block。

典型写入目录：

```text
wiki/sources/      来源文章摘要
wiki/entities/     实体：平台、公司、工具、客户、品牌等
wiki/concepts/     概念：方法论、指标体系、策略框架等
wiki/synthesis/    综合结论和跨来源沉淀
wiki/queries/      从 Chat 或研究结果保存的问题页
wiki/index.md      全局索引
wiki/overview.md   当前 Wiki 概览首页
wiki/log.md        入库和维护日志
```

企业使用时应把 `raw/sources/` 理解为“档案室”，把 `wiki/` 理解为“正式知识库”，把 `review` 理解为“人工确认队列”。

## 资料状态与重复提取

当前系统使用 `.llm-wiki/ingest-cache.json` 记录已成功 ingest 的原始资料。

判断逻辑：

- 原始文件名相同。
- 原始内容 hash 没变化。
- 上次生成的 Wiki 文件仍然存在。

满足这些条件时，资料可被标记为“已提取”。如果内容变化、缓存不存在、或生成页被删除，则标记为“新资料”或“需要重新提取”。

企业版交互建议：

```text
新资料       -> 显示“提取到 Wiki”
检查中       -> 禁止重复点击，等待状态刷新
已提取       -> 显示“已提取”，默认禁用重复 ingest
内容已变更   -> 显示“可重新提取”
失败         -> 显示“重试”
```

不建议隐藏已提取资料。原始资料需要保留，用于追溯来源、删除来源时同步清理相关 Wiki 页面，以及人工复查。

## 知识图谱逻辑

知识图谱不是一套独立知识库，而是从 `wiki/` 页面实时构建出来的结构视图。

图谱节点：

- 一个 Markdown 页面就是一个节点。
- 节点类型来自 frontmatter 的 `type`，例如 `source`、`entity`、`concept`、`synthesis`、`overview`。
- 节点大小通常反映连接数量，即该页面在知识网络中的重要程度。

图谱边：

- 页面正文里的 `[[wikilink]]` 会形成显式连接。
- 页面共享同一 `sources` 来源会形成相关性信号。
- 共同邻居、类型亲和度、直接链接会影响边权重。

图谱社区：

- 系统使用社区发现算法将相关页面聚成知识簇。
- 对外贸电商场景，可能形成“领英获客”“Facebook 广告”“AI 工作流”“客户询盘”“合规与风控”等簇。

图谱的作用：

- 帮管理者看知识是否已经形成体系。
- 发现孤立页面、薄弱主题和关键桥接页。
- 辅助后续 Chat 检索扩展，把强相关邻居页面一起纳入上下文。
- 作为维护提示来源，帮助负责人补链、合并、扩写或标记过时内容。

图谱右侧的 `Insights` 可以理解为 Wiki 健康检查：

- `Surprising Connections`：跨社区、跨类型但存在联系的页面，可能是有价值的新关联。
- `Knowledge Gaps`：孤立页面、稀疏知识簇、关键桥接节点等，需要人工维护。

## Chat / RAG / Rerank 逻辑

当前 `llm_wiki` 的 Chat 是基于当前 Wiki 项目回答问题，不是直接把所有原始资料塞给模型。

实际流程：

```text
用户问题
  -> 读取 wiki/index.md 和 purpose.md
  -> 在 wiki/ 页面中做关键词和短语检索
  -> 同时搜索 raw/sources/ 作为补充召回
  -> 如果 embedding 开启，合并向量召回结果
  -> 构建知识图谱并扩展相关邻居页面
  -> 按上下文预算选择页面
  -> 将页面内容作为 RAG 上下文发给 LLM
  -> LLM 基于这些页面回答，并用编号引用来源
```

当前召回信号包括：

- 文件名精确匹配。
- 标题短语匹配。
- 正文短语出现次数。
- 中文分词和 bigram token 命中。
- embedding 相似度结果。
- 图谱邻居扩展。

当前 `llm_wiki` 前端 Chat 的排序更接近：

```text
规则打分 + 可选向量召回加权 + 图谱扩展排序
```

主 FastAPI 后端已经提供 `/api/llm-wiki/rerank` 代理配置入口，但当前 `llm_wiki` Chat 尚未真正调用独立 rerank 模型进行二次重排。

理想企业版下一步：

```text
关键词召回 topN
  + 向量召回 topN
  + raw/sources 补充召回
  -> 去重
  -> Rerank 模型二次排序
  -> 取 topK 页面
  -> 图谱扩展补充上下文
  -> LLM 回答
```

企业使用边界：

- 已 ingest 并审核过的 `wiki/` 页面应作为优先证据。
- `raw/sources/` 可作为补充证据，但不应压过正式 Wiki。
- 如果回答来自未审核来源，应在 UI 或回答里提示“来自原始资料，尚未沉淀”。
- Rerank 应由主 FastAPI/runtime_config 统一配置，前端不暴露密钥。

## 版本与过期机制

企业知识不应该简单覆盖旧内容。

建议规则：

- 页面正文展示当前有效结论。
- 页面底部保留变更记录。
- 重大变化生成版本摘要或历史段落。
- 所有变化都保留来源链。
- LLM 可以提示过时和冲突，但重大改动进入审核队列。

页面示例结构：

```markdown
# 某客户订单发货延迟

## 当前结论

...

## 背景

...

## 处理过程

...

## 后续预防

...

## 来源

- 飞书消息：om_xxx
- 附件：delivery-note.pdf

## 变更记录

- 2026-04-24：创建页面。
- 2026-04-26：补充复盘结论。
```

## 定时任务建议

第一版可以采用固定批处理。

每日任务：

- 读取昨日或当日新增飞书消息。
- 按群/业务域分组。
- 使用对应 prompt 提炼候选卡片。
- 进行敏感信息初筛。
- 写入审核队列。

每周任务：

- 汇总已批准内容。
- 更新 `overview.md`。
- 提示孤立页面、重复页面、缺失页面。
- 生成流程/FAQ/项目周度更新建议。

每月任务：

- 检查过时内容。
- 合并重复页面。
- 生成管理层月度知识摘要。
- 提示长期未审核候选项。

## Mac mini 本地化部署

该方向适合中小企业的 Mac mini 生态。

推荐部署：

- Mac mini 作为主节点，运行飞书服务、数据库、任务队列和本地模型。
- NAS 保存备份、原始附件、Wiki 快照和模型文件。
- 多台 Mac 可作为推理节点，按任务类型分担 LLM、embedding、rerank。
- 现有 `launchd`、备份、诊断机制继续作为基础运维能力。

优势：

- 数据本地化。
- 成本可控。
- 扩容灵活。
- 适合中小企业渐进式建设。

风险：

- 单点故障需要 NAS 和离机备份缓解。
- 夜间批处理可能堆积，需要任务队列和失败重试。
- 模型效果会影响候选质量，需要持续调 prompt 和抽检。

## MVP 范围

第一版建议只做最小闭环。

包含：

- 复用现有消息、附件、解析内容和权限数据。
- 每日按群生成候选知识卡片。
- 支持 4 类候选：`decision`、`project_update`、`issue`、`faq`。
- 支持人工批量审核。
- 审核通过后写入 Markdown Wiki。
- 页面保留来源、状态、敏感等级和变更记录。
- 每周运行一次 Wiki lint，提示重复、冲突、过时和孤立页面。

暂不包含：

- 全自动正式发布。
- 复杂细粒度页面权限。
- 多人实时编辑。
- 完整图数据库。
- 全量历史聊天一次性重建。

## 后续讨论重点

下一步需要继续细化：

- 企业 Wiki 项目目录如何放置。
- 候选知识表结构如何设计。
- 审核页面如何呈现。
- 每类群的 prompt 如何编写和管理。
- 正式 Wiki 页面如何被 Web UI 或 Obsidian 使用。
- 是否需要独立 `wiki_search`，而不是扩展现有 `/api/query` 混查原始层和 Wiki 层。
- 如果未来需要 Wiki 页面向量，应使用独立向量库或独立命名空间，并先经过产品评审。

当前阶段明确不讨论：

- 正式 Wiki 页面自动发送到飞书 Bot。
- Wiki 页面反向进入主 RAG。
- Wiki 页面向量与原始 chunk 向量直接协同。

这份方案的核心判断是：

> 现有项目继续做企业事实底座，企业 Wiki 作为 LLM 驱动的知识编译和治理层，在人工审核后沉淀为正式组织知识。
